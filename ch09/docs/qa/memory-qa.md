# ch09 答疑：记忆提取与注入 + LLM 选择的安全性

> 日期：2026-06-19

---

## Q1: 记忆的提取和注入细节是怎样的？

记忆有四次流转：

**① 提取（auto_memory）**：每 5 轮触发，`asyncio.ensure_future` 后台异步执行，不阻塞主循环。
- 取增量对话（只取上次提取之后的新消息）
- 调 LLM 输出完整 memories.md（非追加，LLM 可自行去重/合并/删旧）
- 按分类拆写两个文件（用户级 vs 项目级）

**② 扫描（recall）**：扫描 `.mewcode/memory/` 下所有 `.md` 文件，只读前 30 行拿 YAML frontmatter。

**③ 选择（recall）**：LLM 看文件名+描述，返回最多 5 个文件名。白名单校验。

**④ 注入（recall）**：读完整文件内容，包装成 `<system-reminder>`，`conversation.add_system_reminder()` 插入对话。

## Q2: 用 LLM 选最多五条记忆不会出问题吗？

会出问题，但代码做了多层防御：

| 防御 | 机制 | 效果 |
|------|------|------|
| 白名单校验 | `valid_filenames` — LLM 返回的文件名必须在扫描列表中 | 防 LLM 胡编 |
| 静默降级 | LLM 调用失败/超时/返回非 JSON → 全部返回 `[]` | 不影响主对话 |
| 超时保护 | `asyncio.wait_for(prefetch_task, timeout=3.0)` | 3 秒不阻塞 |
| 去重 | `already_surfaced` 集合追踪已展示 | 不重复提醒 |
| 时效性警告 | >1 天的记忆自动标注"可能过时" | 防过期信息误导 |

**三个已知问题**：
1. LLM 可能漏选（描述写得不好时）→ 宁可漏选也不错选
2. LLM 可能错选 → 成本很低（多占几百 token）
3. 没有关键词匹配兜底（Claude Code 原版有）→ 记忆是锦上添花，失败了等下一轮

**核心态度**：记忆是锦上添花，不是必需品。任何一步失败 = 静默降级，不动主对话。
