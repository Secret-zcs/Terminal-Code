# MewCode 模拟面试全程实录

> 日期：2026-07-18
> 角色：大厂后端/Agent 开发面试官
> 形式：追问式深度面试，逐题评价 + 标准答案

---

## Q1: 项目介绍

**面试官**：介绍一下这个项目，3分钟以内。

**候选人**：MewCode 是我学习 Agent 开发时仿真 Claude 做的轻量级代码智能体，模拟了上下文管理、记忆管理、权限控制、工具管理等 Harness 设计，同时引入 Hermes 自进化机制让 Agent 将复杂易错任务落地成 skill。

**评价**：概括性强，但缺少技术亮点。建议补一句核心架构（"支持 Anthropic/OpenAI/OpenAI Compat 三协议，15 个子系统模块化"）。

---

## Q2: 上下文管理细节 + snip/persist/compact 三层

**面试官追问**：snip 触发条件、落盘阈值、保留窗口计算、compact 阈值怎么定的。

**候选人回答**：snip 50轮触发并落盘；persist 给 2000 字符预览；compact 保留最近 5 次工具调用 + 项目进度文档；阈值 = 窗口 - 20K 摘要 - 13K 安全边距。

**评价**：
- ❌ snip 50 轮记混了（代码里是 10 轮），且 snip 不落盘，只留 200 字 `<snipped>` 预览
- ❌ 保留窗口是 Token 预算 + 最少 5 条消息 + tool pair 对齐，不是"5 次工具调用"
- ❌ "可复用结果加标签"代码里不存在
- ✅ 167K 阈值公式正确，但没说出 13K 的构成

**标准答案要点**：
- Layer1 Pass1(单条>50K落盘)、Pass2(合计>200K从大开始砍)、Pass3(超过10轮裁剪)
- Layer2 保留窗口：10K token + 最少5条 + 对齐 tool_use/tool_result 对
- 熔断器：连续 3 次失败禁止自动压缩
- 13K 构成：System prompt(~3K) + 工具定义(~2K) + 摘要 prompt(~1K) + Token 估算误差(~5K) + 杂项(~2K)
- Design B：Layer1 操作在副本 api_conv 上，不修改原始 conversation

## Q3: compact 失败的兜底机制

**面试官追问**：压缩 LLM 调用失败怎么办？死循环怎么防？

**候选人回答**：超时+重试，重试一定次数还失败就返回错误；摘要质量用模型打分，不够分重新生成。

**评价**：
- ✅ 重试 + 超时思路正确
- ⚠️ 打分模型有成本问题（两次 LLM 调用），便宜方案是结构化输出强制校验 9 个必填段
- ⚠️ 没提熔断器（Circuit Breaker）——连续失败后禁止自动压缩

**标准答案**：熔断器 + 渐进式重试（prompt too long 时删 20% 旧轮次）+ 早期退出（前缀<2K不压缩）。

## Q4: 记忆提取的污染风险

**面试官追问**：LLM 提取错误信息并持久化→污染后续对话怎么办？

**候选人回答**：模型决策后用户审批 + 多次测试成功才申请应用。

**评价**：
- ✅ 用户审批 + 测试验证双重保护
- ✅ 知道临时错误不该固化

**标准答案补充**：
- 自动回归验证：用创建时的对话快照做回归测试，三次变体 prompt 都通过 → 待审批
- 版本回退：新版本回归失败自动回退
- 独立模型审核：提案先给另一个模型判断"是否临时场景"

## Q5: 多 Agent 任务分解

**面试官追问**："调研 Python Web 框架优缺点"怎么分解？子 Agent 卡死怎么发现？

**候选人回答**：串行分解（调研→分析→对比），子 Agent 完成主动通知 + 长时间无响应主动询问，超时无响应中断。

**评价**：
- ✅ 混合模式（被动接收+主动询问）设计好
- ⚠️ 调研阶段可并行（Django/FastAPI/Flask 互不依赖），串行浪费 3 倍时间
- ⚠️ Mailbox 消息在子 Agent 卡死时收不到→心跳检测需共享内存时间戳

**标准答案**：三路并行调研 → 一路汇总对比。心跳用 `time.monotonic()` 共享内存时间戳而非 Mailbox 消息。任务表 + 版本号防过期结果覆盖。

## Q6: 子 Agent Task 分配和崩溃恢复

**面试官追问**：认领任务后、写 agent_id 前崩溃→任务蒸发。取消 Agent 时写了一半的文件怎么办？

**候选人回答**：重新创建 Agent。取消时不知道怎么办。

**评价**：
- ✅ 重新创建大方向对
- ❌ 任务蒸发的根因没抓到——认领不是原子操作

**标准答案**：
- 任务表（非 Mailbox 文件队列）：status 字段 → claimed 超时回退为 pending
- 版本号：Agent 提交结果时校验，旧版本被拒绝
- 取消策略：标记位 `_cancelled` 让 Agent 自然退出（不等的话用 tmp 文件 + `os.replace` 原子写入）

## Q7: 并发工具调用的数据竞争

**面试官追问**：同轮 ReadFile + EditFile 同一文件→读到的内容是旧还是新？代码能保证什么？

**候选人回答**：访问同一文件加锁；看谁先来；保证只有协程执行。

**评价**：
- ❌ 代码里没有文件级别的锁
- ✅ 实际靠 partition_tool_calls() 的分区+串行批保证顺序

**标准答案**：is_concurrency_safe=False 的工具独自成批，批之间严格串行。语义依赖由 LLM 在轮次级别控制，执行顺序由分区算法在轮内部控制。

## Q8: Task 工具 vs TodoWrite

**面试官追问**：TaskCreate/TaskGet/TaskList/TaskUpdate 跟 Claude Code 的 TodoWrite 什么关系？Task 依赖和分区执行怎么联动？

**候选人回答**：我不太懂。

**标准答案**：
- 不是一回事：TodoWrite = 单 Agent 内部计划，TaskCreate 系列 = 跨 Agent 共享任务表
- 不联动：Task 依赖 = 任务调度层（跨轮次），工具分区 = 工具执行层（同轮内），各司其职

---

## Q9: 多协议适配为什么用 ABC 继承

**面试官追问**：三种协议行为不同，为什么选 ABC 继承而不是一个大函数加 if-else？

**候选人回答**：我不太懂。

**标准答案**：三点——开闭原则（新增协议=新增子类，不改已有代码）、统一输出（StreamEvent 接口隔离协议差异）、各自状态管理（Anthropic 需要 thinking_signature，OpenAI 需要 active_calls，互不干扰）。

---

## Q10: 三人分工方案

**内容**：阶段一自己写最小链路+锁接口，阶段二三人并行（你核心链路、A上下文记忆、B权限工具），阶段三串行集成高级子系统。

**标准答案要点**：接口锁定是并行的前提条件。按耦合度分配而非平均分工。集成从第二周就开始，不等最后一次性。
